# /spec-clarify
> spec 초안 완성 후 소크라테스식 7문항 검증 — 과도한 설계·범위 충돌·가정 오류 조기 차단

## 트리거
- `/spec-clarify` — 현재 세션의 spec 초안을 대상으로 실행
- `/spec-clarify [도메인] [버전]` — 명시적 대상 지정
- spec-define Step 5.5 완료 후 자동 호출

---

## 범위

하는 것: spec 초안 7문항 검증 + PASS/WARN/FAIL 판정 반환
하지 않는 것: spec 작성, /isok 실행, /vs 실행, TaskCreate, 파일 직접 수정

## 전제 조건 (가정)

1. spec 초안이 세션 내에 있거나, 지정 파일 경로에 존재한다
2. spec-define Step 5.5 이후 호출 시 PRD/plan 파일이 생성된 상태다
3. UNRESOLVED 항목은 사람이 직접 검토한다 — 자동화 세션 시 텔레그램으로 전송

## 실행 원칙

⛔ **인라인 시뮬레이션 금지. 실제 spec 파일 또는 세션 내 초안을 읽고 검증.**
⛔ **7문항 전부 실행. 임의 스킵 불가.**
⛔ **답변이 불명확한 문항은 UNRESOLVED로 표시 — 사람에게 확인 요청.**

---

## 검증 순서

### 1. spec 초안 로드
세션 내 초안이 있으면 그것 사용.
없으면 아래 순서로 파일 로드:
- `obsidian/03_Projects/[도메인]/[버전]-PRD.md`
- `obsidian/03_Projects/[도메인]/[버전]-plan.md`
- 도메인 spec 파일

초안·파일 모두 없을 경우: 즉시 중단 + "검증 대상 spec 없음. spec 파일 경로나 초안을 제공해 주세요." 출력 후 종료.

### 2. 소크라테스 7문항 실행

각 문항마다: **질문 → 초안에서 발견한 답 → 판정 (✅/⚠️/❌)**

| # | 문항 | 판정 기준 |
|---|------|----------|
| Q1 | **범위**: 이 spec이 무엇을 하는가? 무엇을 하지 않는가? | Out-of-Scope 명시 여부 |
| Q2 | **용어**: 핵심 용어가 명확히 정의되었는가? | 모호한 용어 = ⚠️ |
| Q3 | **가정**: 이 설계가 전제하는 것들은 무엇인가? | 미검증 가정 = ⚠️ |
| Q4 | **성공 기준**: 어떤 상태면 완료인가? 측정 가능한가? | 수치 없음 = ❌ |
| Q5 | **엣지케이스**: 실패·예외 케이스가 정의되었는가? | 누락 = ⚠️ |
| Q6 | **의존성**: 다른 시스템·파일·사람에 의존하는가? | 미명시 의존성 = ⚠️ |
| Q7 | **충돌**: 기존 CLAUDE.md·설계 규칙과 충돌하는가? | 충돌 발견 = ❌ |

판정 출력 형식:
```
## /spec-clarify 검증

Q1 [범위] ✅
  무엇을: ...
  하지 않는 것: ...

Q2 [용어] ⚠️
  모호한 용어: "버전업" — "기능 추가 vs 구조 변경" 구분 불명확
  → 제안: "구조 변경을 포함하면 spec-hardening 필수" 명시 권장

Q3 [가정] ✅
  전제: ...

Q4 [성공 기준] ❌
  현재: "spec 완성 후 ok" (측정 불가)
  → UNRESOLVED: "어떤 수치/상태면 완료인지 명확히 정의 필요"

Q5 [엣지케이스] ⚠️
  누락: Q&A 미답변 시 흐름 미정의
  → 제안: "48시간 무응답 시 draft 저장 후 중단" 추가

Q6 [의존성] ✅
  ...

Q7 [충돌] ✅
  ...
```

### 3. 판정 집계

```
## 검증 결과 요약

✅ 통과: N개
⚠️ 경고: N개
❌ 실패: N개
UNRESOLVED: N개

판정: [PASS / WARN / FAIL]
```

판정 기준:
- **PASS**: ❌ 0개, UNRESOLVED 0개 (⚠️는 기록만)
- **WARN**: ❌ 0개, UNRESOLVED 1개 이상 → 사람 확인 후 진행 가능
- **FAIL**: ❌ 1개 이상 → spec 수정 후 재실행 필수

### 4. 분기

**PASS / WARN:**
```
spec-clarify 완료. 다음 → /isok 실행 (spec-define Step 6)
WARN 항목은 isok 전 해소 권장.
```

**FAIL:**
```
❌ 항목 수정 필요. spec 초안 업데이트 후 /spec-clarify 재실행.
재시도 최대 2회. 2회 후에도 FAIL → 사람에게 중단 보고.
```

## 독립 실행 시

spec-define 외부에서 단독으로 호출 가능.
- 대상 spec 파일 경로를 명시하거나 현재 세션 초안 사용
- 완료 후 PASS/WARN/FAIL 결과 반환 (spec-define 흐름에 미개입)
- 다음 단계 안내: PASS/WARN → "spec 확인 후 /spec-build 또는 /isok 진행 가능" / FAIL → "spec 수정 후 재실행"
