# 기능 인벤토리 & 동선 분석

## STEP 2.5 — 기능 인벤토리 (Feature Inventory)

앱/서비스의 기능을 전수 조사한다. gstack browse가 사용 가능하면 실제 탐색으로 추출, 없으면 검색·앱스토어·공식 사이트로 수집.

---

### ⏰ 최신 기능 탐색 (기능 수집 전 항상 먼저 실행)

기능 인벤토리에서 "이 기능 없음"이라고 쓰면 바로 신뢰가 무너진다.
서비스는 계속 업데이트된다. 오래된 정보로 분석하면 틀린 분석이 된다.

**[사전 체크] 스크린샷 여부 확인 — 분석 시작 직전에 묻는다**

분석 대상이 모바일 앱이고 직접 앱 접근이 불가한 경우,
분석을 시작하기 전 사용자에게 가볍게 먼저 물어본다:

```
"[서비스명] 앱 분석을 시작할게요.
 혹시 지금 앱 스크린샷 몇 장 찍어서 올려주실 수 있나요?
 (홈 화면, 하단 탭 바, 설정/마이페이지 — 3장이면 충분해요)
 있으면 기능 분석이 훨씬 정확해지고, 없어도 검색으로 진행합니다."
```

→ 사용자가 "없어도 돼요", "바로 해줘" 하면 즉시 검색으로 진행.
→ 사용자가 스크린샷을 올리면 그 이미지를 기능 인벤토리 근거로 활용.
→ 이 질문은 딱 한 번만, 분석 시작 전에만 한다. 분석 중간에 끊지 않는다.

---

**탐색 순서 (검색 기반):**

```
① 공식 블로그 / 테크 블로그 검색  ← 가장 정확한 소스
   검색어: "서비스명 새기능 [현재연도]"
           "서비스명 업데이트 [현재연도]"
   URL 패턴: about.서비스명.com, 서비스명.tech, tech.서비스명.com,
             blog.서비스명.com, 서비스명.tistory.com 등
   → 직접 출시한 기능은 여기서 가장 먼저 발표됨

   ⚠️ 주의: "서비스명 업데이트"로 검색 시 B2B 개발자 문서가 섞일 수 있다.
   소비자 앱 기능을 찾을 땐 "서비스명 앱 기능"이나 "서비스명 사용자" 키워드 추가.

② 앱스토어 릴리즈 노트
   검색어: "서비스명 앱스토어 새로운기능 [현재연도]"
           "서비스명 app store release notes [현재연도]"
   → 버전별 변경사항이 가장 정확하게 기록됨

③ 테크 미디어 최신 기사 (최근 6개월)
   검색어: "서비스명 신기능 출시 [현재연도]"
           "서비스명 앱 기능 추가 [현재연도]"
   → 주요 기능은 미디어에도 커버됨. 부제목·소제목까지 확인.

④ [선택] 스크린샷·영상 기반 탐색
   검색어: "서비스명 앱 스크린샷 [현재연도]"
           "서비스명 앱 리뷰 [현재연도]"
   → 텍스트로 안 나온 UI 변경 사항 포착 가능
```

**탐색 후 결과 반영:**
- 최근 6개월 내 출시 기능은 🆕 표시
- 탐색 과정에서 확인한 기능은 출처 명시 (예: "2025.09 보도자료 기반")
- 탐색 결과 없으면 "최근 변경 없음 (확인일: [현재날짜])" 명시

### 기능 수집 방법

**gstack browse 사용 가능 시 (URL 있을 때):**
```bash
# 1. 메인 화면 탐색
$B goto <서비스 URL>
$B snapshot -i          # 전체 네비게이션 구조 파악
$B screenshot /tmp/main.png

# 2. 주요 메뉴 순회
# GNB 각 항목 클릭 → 화면 캡처 → 기능 추출

# 3. 설정·마이페이지 탐색 (숨겨진 기능 발굴)
$B goto <설정 URL>
$B snapshot -i

# 4. 온보딩 플로우 추적
$B goto <회원가입 URL>
$B snapshot -D          # 각 단계 diff로 흐름 추적
```

**URL 없을 때:**
- 앱스토어 설명 + 스크린샷 텍스트 추출
- 공식 랜딩페이지 메뉴 구조
- 유튜브 리뷰·튜토리얼 영상 설명
- 블로그/미디엄 사용기 분석

## STEP 2.6 — 동선 분석 (User Flow Analysis)

4개의 핵심 플로우를 분석한다. 각 단계별 마찰 지점을 찾는 것이 목적.

### 분석할 4대 플로우

#### 1. 온보딩 플로우 (첫 경험)
```
목표: 신규 가입자가 첫 핵심 가치를 경험하기까지의 경로

분석 항목:
- 가입 단계 수 / 필수 입력 항목 수
- 첫 핵심 액션까지 걸리는 단계
- "아하 모멘트" 도달 시간 추정
- 이탈 가능 지점 (선택 항목, 장벽)

마찰 기준:
🔴 3분 내 핵심 가치 미경험
🟡 가입 단계 5단계 이상
🟢 1분 내 핵심 가치 경험 가능
```

#### 2. 핵심 액션 플로우 (Core Loop)
```
목표: 서비스의 핵심 기능을 반복 사용하는 사이클

분석 항목:
- 핵심 액션이 무엇인가 (DAU를 만드는 행동)
- 한 사이클 완료에 걸리는 단계
- 진입 경로 수 / 완료 후 다음 액션 유도 여부

마찰 기준:
🔴 핵심 액션 진입 3단계 이상
🟡 완료 후 다음 행동 유도 없음
🟢 홈에서 1단계로 핵심 액션 진입
```

#### 3. 전환 플로우 (무료 → 유료)
```
목표: 유료 결제를 유도하는 경로

분석 항목:
- 페이월 등장 시점 (어떤 행동을 하면 나오나)
- 유료 전환 CTA 위치·빈도 / 결제 완료까지 단계 수
- 무료 체험 유무 및 기간

마찰 기준:
🔴 결제 단계 5단계 이상
🟡 페이월이 너무 이르거나 너무 늦음
🟢 자연스러운 페이월 + 3단계 이내 결제
```

#### 4. 재방문 플로우 (Retention Hook)
```
목표: 사용자가 다시 돌아오게 만드는 장치

분석 항목:
- 푸시/이메일 알림 유무 및 적절성
- 스트릭·연속 기록 등 습관 형성 장치
- 미완성 작업 유도 알림 / 소셜 요소

마찰 기준:
🔴 재방문 유도 장치 없음
🟡 알림 있으나 관련성 낮음
🟢 맥락 기반 개인화 알림
```

## Nielsen 10 휴리스틱 빠른 체크 (DEEP 모드)

| # | 휴리스틱 | 체크 포인트 |
|---|---------|-----------|
| 1 | 시스템 상태 가시성 | 로딩·진행상황이 보이는가 |
| 2 | 실제 세계와 일치 | 사용자 언어로 쓰여 있는가 |
| 3 | 사용자 통제감 | 실행 취소·뒤로가기 가능한가 |
| 4 | 일관성 | 같은 기능이 같은 방식으로 작동하는가 |
| 5 | 오류 예방 | 실수하기 어려운 구조인가 |
| 6 | 기억 부담 최소화 | 외울 필요 없이 보면 알 수 있는가 |
| 7 | 유연성·효율성 | 숙련 사용자를 위한 단축 경로 있는가 |
| 8 | 미적 단순성 | 불필요한 정보가 없는가 |
| 9 | 오류 복구 | 오류 메시지가 명확하고 해결책 제시하는가 |
| 10 | 도움말 | 설명 없이도 쓸 수 있는가 |
